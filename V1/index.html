V1 / Practical Workflow

AI 赋能游戏开发 从“手工做”到“带 Agent 做”

这版少讲概念，多讲一个游戏从策划、UI、逻辑、资源、测试到构建，在 Cocos Creator 和 Unity 里怎么落地。

01 / 先用一个例子开场

我们假设要做一个 2D 休闲小游戏

例子：竖屏“躲避 + 收集”小游戏

  • 玩家左右移动角色，躲开障碍，收集金币。
  • 60 秒一局，结算分数，可重新开始。
  • 有主界面、游戏 HUD、暂停、结算界面。
  • 目标不是做大作，而是展示完整研发闭环。

分享时重点讲什么

  • 策划案怎么写，才能让 AI 真的能执行。
  • UI 以前手动点，现在如何让 Agent 生成和搭建。
  • 游戏逻辑如何和编辑器里的 UI / Prefab 绑定。
  • 测试反馈从哪里来：日志、截图、自动测试、构建报告。

先让听众跟着一个小项目走，比一上来讲“AI 原生范式”更容易理解。

02 / AI 开发历程，用一句话讲清

从补全代码，到把一段研发流程交给 Agent

阶段 1

代码补全

AI 补一行代码、一个函数。人还是主要执行者。

阶段 2

聊天问答

把报错、需求、算法拿去问，AI 返回解释和示例。

阶段 3

流程加 AI

策划、代码、美术、测试、运营素材，各环节都有 AI 工具。

阶段 4

Agent 跑任务

AI 读工程、改文件、跑命令、看日志、修失败，再给人验收。

人的角色变成：定目标、给约束、看结果、做取舍；AI 的角色变成：执行、试错、整理反馈。

03 / 工具只讲够用的

这类 Agent 工具在游戏项目里能做什么

工具
直接能力
游戏项目里的用法
Claude Code
读代码库、编辑文件、运行命令、集成开发工具。
改 Cocos / Unity 脚本、生成测试、修构建错误、批量重构。
Gemini CLI
开源命令行 AI Agent，适合终端、批处理和大段上下文分析。
分析日志、批量处理配置、写资源导入脚本、整理设计文档。
GitHub Copilot
IDE 内补全、聊天、Agent mode、PR 辅助。
日常编码、补测试、解释旧代码、帮助新人理解模块。

工具不是重点。重点是给它“工程上下文 + 明确验收方式”，否则就只是聊天。

04 / 策划案怎么写

给人的策划案，改成给 Agent 也能执行的规格

以前常见写法

做一个简单躲避游戏：
玩家躲避障碍，吃金币加分。
有开始和结束界面。
手感要轻快。

问题：边界太模糊，AI 会自己脑补速度、碰撞、UI、分数、失败条件。

适合 Agent 的写法

目标：Cocos Creator 3.8，竖屏 720x1280。
核心循环：60 秒内左右移动角色，收集 Coin +10，撞到 Bomb -1 HP。
失败：HP=0 或倒计时结束。
UI：StartPanel、HudPanel、PausePanel、ResultPanel。
输入：PC 用 A/D，手机用左右触摸区域。
验收：能在预览里跑 1 局；Console 无 error；截图包含 HUD 分数和倒计时。

05 / Cocos Creator 第一步

先让 Agent 建“可运行骨架”，不要先追求完整美术

1

把规格放进项目

创建 docs/game-spec.md：玩法规则、场景列表、UI 节点命名、资源目录、验收命令。

2

让 Agent 生成 TypeScript 结构

例如 GameManager、PlayerController、Spawner、ScoreModel、UIBinder。

3

先用占位资源跑通

玩家是蓝色方块、金币是黄圈、炸弹是红圈。先验证循环，再替换美术。

4

每一轮都读 Console

Cocos Creator Console 会显示 log / warn / error，还可以输出可定位 asset、node、script 的信息，作为 Agent 修复反馈。

06 / Cocos Creator UI

以前手点 UI，现在把 UI 变成“可生成的节点规格”

以前怎么做

  • 在层级面板里手动创建 Canvas、Panel、Label、Button。
  • 手动拖锚点、改坐标、绑定按钮事件。
  • 每改一次 UI，程序和策划要反复对节点名。

AI 工作流怎么做

  • 先写 ui-spec.json：面板、节点名、组件、布局、点击事件。
  • Agent 生成 Cocos Creator Editor 扩展或场景脚本，批量创建节点。
  • 用固定节点名绑定代码：HudPanel/ScoreLabel、ResultPanel/RestartButton。

核心变化：不是“AI 替你点鼠标”，而是把鼠标操作抽象成 JSON / 脚本，让它可重复生成、可审查、可回滚。

07 / Cocos UI Prompt 示例

可以直接给 Agent 的 UI 搭建任务

请基于 Cocos Creator 3.8 创建 UI 搭建脚本。
输入：docs/ui-spec.json
输出：
1. 生成 Main.scene 的 Canvas/UIRoot。
2. 创建 StartPanel、HudPanel、PausePanel、ResultPanel。
3. 所有节点使用规范命名，方便脚本 find。
4. Button 只绑定到 UIBinder 暴露的方法名，不写业务逻辑。
5. 运行后在 Console 输出每个创建节点的路径。
验收：
- 场景层级里能看到 4 个面板。
- Console 无 error。
- 重新运行脚本不会重复创建同名节点。

给 Agent 的关键约束

  • 只做 UI 结构，不写玩法逻辑。
  • 所有节点名稳定，后续脚本依赖这些名字。
  • 按钮事件只连到中间层，避免 UI 和玩法强耦合。
  • 日志必须可定位，方便下一轮反馈。

08 / Cocos 核心逻辑

游戏逻辑不要绑死在 UI 上：用事件和数据模型连接

Model

GameState

score、hp、timeLeft、gameStatus。纯数据，方便测试。

Logic

GameManager

开始、暂停、恢复、结算、重开；派发状态变化事件。

Actor

Player / Item

移动、生成、碰撞、回收；尽量不直接改 UI。

Bridge

UIBinder

监听状态变化，把 score/time/hp 写到 Label。

Config

Balance

速度、掉落间隔、分数、血量放 JSON，策划可改。

给 Agent 的任务可以拆成：“先实现纯逻辑 + 单测”，再“接入 Cocos 组件和 UI”。这样更容易定位错误。

09 / Cocos 测试与反馈

以前靠人点，现在让 Agent 读反馈再修

可以自动化的反馈

  • Console 日志：error、warn、节点创建记录、资源路径。
  • 构建日志：Cocos Creator 支持命令行构建，适合接 CI。
  • 截图对比：固定分辨率预览，检查 HUD 是否出现、文本是否溢出。
  • 逻辑测试：把分数、倒计时、碰撞结果抽成纯 TS 函数，用 Jest/Vitest 测。

反馈给 Agent 的格式

本轮结果：
1. Preview 能启动。
2. 点击 Start 后 Console 报错：
   Cannot read property 'string' of null
3. 截图显示 HudPanel 出现，但 ScoreLabel 为空。
4. 期望：
   - 修复节点查找。
   - 增加启动时节点校验日志。
   - 不要改 ui-spec.json。

10 / Unity 同样展开

Unity 里也按“规格 → 编辑器脚本 → 逻辑 → 测试”走

1

策划案转 ScriptableObject 规格

把血量、速度、掉落概率、关卡时长放到 GameConfig.asset，策划改数据，程序不改代码。

2

用 Editor 脚本生成场景

Agent 写 MenuItem 或 EditorWindow：创建 Canvas、Prefab、HUD、按钮事件、测试场景。

3

业务逻辑和 MonoBehaviour 分层

纯 C# 类处理规则；MonoBehaviour 只负责输入、生命周期、渲染、音效和 UI 绑定。

4

用 Test Framework 验收

EditMode 测纯逻辑和编辑器工具；PlayMode 测运行时行为、碰撞、UI 状态。

11 / Unity UI 示例

让 Agent 写 EditorWindow，比让它描述“怎么点”更可靠

请在 Unity 2022/2023 项目中创建工具：
Assets/Editor/PrototypeSceneBuilder.cs

菜单：Tools/AI Prototype/Build Dodge Game Scene
执行后：
1. 创建 SampleScene 或清理同名节点。
2. 创建 Canvas、StartPanel、HudPanel、ResultPanel。
3. 创建 Player、CoinPrefab、BombPrefab 占位对象。
4. 自动挂载 GameManager、UIBinder、Spawner。
5. 把 Button.onClick 连接到 GameManager.StartGame / Restart。
6. 保存场景并输出 Editor.log 可读的构建摘要。

为什么这比手动点快

  • UI / Prefab / 事件绑定能反复生成。
  • 每次改规格，重新跑工具就能得到一致结构。
  • Agent 可以直接改 Editor 脚本，而不是猜测你点了什么。
  • 团队可以把工具纳入版本控制和 code review。

12 / Unity 测试反馈

Unity 的反馈从 Console、Editor.log、PlayMode 测试来

EditMode 测试

测试纯 C# 规则、配置加载、Editor 工具是否创建了正确节点。适合跑得快、频繁跑。

PlayMode 测试

进入运行态，模拟几帧或几秒，检查玩家移动、碰撞扣血、结算面板显示。

日志与截图

Unity Editor.log、Player.log、Console 报错、固定视口截图，都能作为 Agent 下一轮修复输入。

人的测试仍然重要，尤其是手感。但“能不能启动、UI 是否绑定、规则是否正确、构建是否失败”可以先交给自动化。

13 / 图片到动画

Gemini 生成图片后，怎么快速变成游戏可用动画

1 图像

Nano Banana 2

生成角色三视图、道具、图标、UI 草案；要求透明背景、统一光照、固定尺寸。

2 拆层

Photoshop / Krita

AI 辅助抠图、补全被遮挡区域，拆成头、身体、手臂、腿、武器等图层。

3 动画

Spine / Rive

做骨骼绑定、循环 idle、walk、hit；Agent 生成命名规范和导出配置。

4 帧图

TexturePacker

把导出的 PNG 序列打成 spritesheet，自动生成 atlas 和 meta。

5 接入

Cocos / Unity

自动生成导入脚本、动画状态机、预览场景和资源校验表。

14 / 动画自动化示例

以前一帧一帧调，现在把重复动作交给脚本和 AI

以前

  • 美术手动导出每一帧。
  • 手动改文件名、拖进引擎、设置 pivot。
  • 程序手动建 AnimationClip / SpriteFrame 动画。
  • 漏帧、命名错、尺寸不统一，经常返工。

现在的 AI 工作流

  • Agent 根据角色规格生成导出清单：idle_00.png 到 idle_11.png。
  • 调用工具批量裁切、统一透明边距、检查尺寸和命名。
  • 自动生成 Cocos SpriteFrame 动画或 Unity AnimationClip。
  • 生成预览场景，自动截图给美术和策划验收。

视频生成工具也可做“动作参考”，但进游戏前通常还要转为骨骼动画、序列帧或可控状态机。

15 / 一人团队与微团队

AI 经验会让角色边界变模糊

策划

不只是写文档，也能写可执行规格、改配置、生成测试用例、做可跑原型。

程序

不只是写功能，也能生成 UI 草图、资源导入工具、自动化测试和构建流水线。

美术 / TA

不只是出图，也能定义资源规范、批量检查素材、生成导出脚本和预览场景。

微团队的核心不是“一个人替代所有人”，而是每个人都能用 Agent 做出第一版，专家再负责关键质量。

16 / 可以落地的团队规范

让 Agent 真正进项目，需要几条硬规则

任务模板

  • 目标：要完成什么，不要完成什么。
  • 上下文：引擎版本、目录、相关脚本、设计文档。
  • 验收：运行命令、截图、日志、测试结果。
  • 边界：不要改哪些文件，不要引入哪些依赖。

质量护栏

  • 小任务、小提交、小 review。
  • 生成资源必须记录来源和授权。
  • 核心战斗、付费、存档、账号逻辑必须人工评审。
  • Agent 只能拿到完成任务所需权限。

17 / 总结

这场分享可以落在三句话

1

AI 不是只帮你写代码

它能参与策划规格、UI 搭建、逻辑实现、资源管线、测试反馈和构建发布。

2

游戏开发最适合“可验证的小闭环”

一个 UI 面板、一个技能、一个关卡、一个资源导入器，都可以变成 Agent 任务。

3

人的价值在判断和组织

懂游戏、懂工程、懂 AI 工作流的人，会成为多个 Agent 的负责人。
